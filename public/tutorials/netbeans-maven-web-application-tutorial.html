Maven Web Application Creation with JavaServer Faces, PrimeFaces, and GlassFish

This tutorial demonstrates how NetBeans creates the basic Maven Web Application including the addition of JavaServer Faces and PrimeFaces

First of all, what exactly is Maven?

Maven is a build automation and project management tool for Java that helps developers organize, compile, test, and package their applications in a consistent way. Instead of manually configuring every step of the build process, Maven uses a standardized project structure and a configuration file called the Project Object Model (POM) to define how the project should be built, what dependencies it needs, and how those dependencies should be managed. The POM file (pom.xml) serves as the blueprint for the entire project, describing everything from the source directory layout to plugins that handle tasks like compiling Java code or creating a .war file for web deployment.

Maven also manages external libraries through its repository system. The Maven Central Repository is a vast public library of precompiled Java packages (JARs) that developers can easily reference in their projects without having to download or configure them manually. When a dependency is listed in the pom.xml, Maven automatically retrieves it—along with any of its required sub-dependencies—from Central (or another configured repository) and stores it in a local cache for future builds. This makes Maven a powerful tool for dependency management, version control, and reproducible builds across different environments

What is Archetype?

"In short, Archetype is a Maven project templating toolkit. An archetype is defined as an original pattern or model from which all other things of the same kind are made. The name fits as we are trying to provide a system that provides a consistent means of generating Maven projects. Archetype will help authors create Maven project templates for users, and provides users with the means to generate parameterized versions of those project templates."

So how does NetBeans use this to create a Maven Web Application?

NetBeans creates the web application using an archetype from the Maven Central Repository. Specifically, NetBeans uses the webapp-jakartaee archetypes from Josh Juneau.

# Create a new Maven Web Application in the current directory.
mvn \
-DarchetypeGroupId=io.github.juneau001 \
-DarchetypeArtifactId=webapp-jakartaee11 \ # use your JarkartaEE version
-DarchetypeVersion=1.0.0 \
-DarchetypeRepository=https://repo.maven.apache.org/maven2 \
org.apache.maven.plugins:maven-archetype-plugin:3.2.1:generate

This command will generate a project in "interactive mode". Interactive mode will allow you do define values for properties required during the creation of the project. For this archetype, you will be asked to define a 'groupID', an 'artifactID', the 'version', and a 'package'.

Example:

Define value for property 'groupId': com.michaelcherrera
Define value for property 'artifactId': Demo
Define value for property 'version' 1.0-SNAPSHOT: :
Define value for property 'package' com.michaelcherrera.Demo: : 
Confirm properties configuration:
groupId: com.michaelcherrera
artifactId: Demo
version: 1.0-SNAPSHOT
package: com.michaelcherrera.Demo
 Y: : Y

Notice that you can leave the 'version' blank. The default will be "1.0-SNAPSHOT". If you leave the 'package' blank, the default will be your groupId with your artifactId. Finally you, you will be asked to confirm your properties configuration. These are same properties you would ordinarily enter into the NetBeans project creation GUI. NetBeans will use your property values to run a command very similar to the mvn command above, except NetBeans will not use interactive mode. Instead NetBeans will enter your property values directly into the command.

What about GlassFish?

Aside from the archetype generation, NetBeans will also create the following file:

<!-- src/main/webapp/WEB-INF/glassfish-web.xml -->
<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE glassfish-web-app PUBLIC "-//GlassFish.org//DTD GlassFish Application Server 3.1 Servlet 3.0//EN" "http://glassfish.org/dtds/glassfish-web-app_3_0-1.dtd">
<glassfish-web-app error-url="">
  <context-root>/<!-- url --></context-root>
  <class-loader delegate="true"/>
  <jsp-config>
    <property name="keepgenerated" value="true">
      <description>Keep a copy of the generated servlet class' java code.</description>
    </property>
  </jsp-config>
</glassfish-web-app>

That file, glassfish-web.xml, is a GlassFish-specific deployment descriptor—essentially a configuration file that customizes how your web application behaves when it’s deployed on the GlassFish application server. While the standard web.xml defines settings according to the Java EE (Jakarta EE) specification—things like servlets, filters, and security roles—glassfish-web.xml adds server-specific options that GlassFish uses during deployment. In short, this XML acts as a bridge between your portable Java web app and the specifics of the GlassFish runtime, ensuring that the app integrates smoothly with that particular server’s environment.

How do we add JavaServer Faces?

In NetBeans, you can easily add JavaServer Faces (JSF) via the project properties GUI. But what exactly is NetBeans doing when you add JSF? To add JSF to your project, NetBeans simply adds settings to the web.xml.

Example:

<!-- src/main/webapp/WEB-INF/web.xml -->
<?xml version="1.0" encoding="UTF-8"?>
<web-app xmlns="https://jakarta.ee/xml/ns/jakartaee"
   xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
   xsi:schemaLocation="https://jakarta.ee/xml/ns/jakartaee https://jakarta.ee/xml/ns/jakartaee/web-app_6_0.xsd"
   version="6.0">
    <context-param>
        <param-name>jakarta.faces.PROJECT_STAGE</param-name>
        <param-value>Development</param-value>
    </context-param>
    <!-- The 'FACELETS_SKIP_COMMENTS' parameter is not added by NetBeans -->
    <context-param>
        <param-name>jakarta.faces.FACELETS_SKIP_COMMENTS</param-name>
        <param-value>true</param-value>
    </context-param>
    <servlet>
        <servlet-name>Faces Servlet</servlet-name>
        <servlet-class>jakarta.faces.webapp.FacesServlet</servlet-class>
        <load-on-startup>1</load-on-startup>
    </servlet>
    <servlet-mapping>
        <servlet-name>Faces Servlet</servlet-name>
        <url-pattern>/faces/*</url-pattern>
    </servlet-mapping>
    <session-config>
        <session-timeout>
            30
        </session-timeout>
    </session-config>
    <welcome-file-list>
        <welcome-file>faces/index.xhtml</welcome-file>
    </welcome-file-list>

    <!-- These MIME mappings are for PrimeFaces
    These are NOT added by NetBeans and should be added manualy -->
    <mime-mapping>
        <extension>otf</extension>
        <mime-type>font/opentype</mime-type>
    </mime-mapping>
    <mime-mapping>
        <extension>ttf</extension>
        <mime-type>application/x-font-ttf</mime-type>
    </mime-mapping>
    <mime-mapping>
        <extension>woff</extension>
        <mime-type>application/x-font-woff</mime-type>
    </mime-mapping>
    <mime-mapping>
        <extension>woff2</extension>
        <mime-type>application/x-font-woff2</mime-type>
    </mime-mapping>
    <mime-mapping>
        <extension>svg</extension>
        <mime-type>image/svg+xml</mime-type>
    </mime-mapping>
    <mime-mapping>
        <extension>eot</extension>
        <mime-type>application/vnd.ms-fontobject</mime-type>
    </mime-mapping>
</web-app>

Context Parameters (<context-param>)

These set configuration values for the application.

  • jakarta.faces.PROJECT_STAGE – Tells JSF which mode the app is running in.

    • Development enables detailed error messages and resource reloading.
    • Production improves performance and hides debugging info.

  • jakarta.faces.FACELETS_SKIP_COMMENTS – Controls whether Facelets template comments (<!-- -->) are included in the rendered HTML.

    • true means comments are removed, reducing page size and hiding internal notes.

Servlet Declaration (<servlet> and <servlet-mapping>)

  • <servlet> – Defines the FacesServlet, which is the core JSF controller. It processes requests, runs the JSF lifecycle, and renders views.

    • <load-on-startup>1</load-on-startup> means it loads when the application starts, not on first request.

  • <servlet-mapping> – Maps URLs to the FacesServlet.

    • <url-pattern>/faces/*</url-pattern> means any request starting with /faces/ will be handled by JSF.

Session Configuration (<session-config>)

  • <session-timeout> – Sets how long (in minutes) a user session can be idle before it expires.
    • Here, 30 means 30 minutes of inactivity.

Welcome File (<welcome-file-list>)

  • Specifies the default page when users access the root of the app.
    • Here, faces/index.xhtml is the landing page.

MIME Mapping (<mime-mapping>)

  • Tells the server what type of content certain file extensions represent.
  • Examples:
    • .ttf → application/x-font-ttf (TrueType font)
    • .woff → application/x-font-woff (Web Open Font Format)
    • .svg → image/svg+xml (Scalable Vector Graphics)

This ensures browsers handle fonts, images, and other resources correctly.

In addition to the web.xml changes, NetBeans creates a new index.xhtml. Extensible HyperText Markup Language is much like it sounds. It makes standard HTML more extensible to work with other data formats (in this case, JSF).

Example:

<?xml version='1.0' encoding='UTF-8' ?>
<!DOCTYPE html>
<html
    xmlns="http://www.w3.org/1999/xhtml"
    xmlns:h="jakarta.faces.html"
>
    <h:head>
        <meta charset="UTF-8" />
        <meta name="viewport" content="width=device-width, initial-scale=1.0" />
        <title>Demo</title>
    </h:head>
    <h:body> </h:body>
</html>

Namespace Declarations in XHTML

  • xmlns="http://www.w3.org/1999/xhtml"

    • Sets the default namespace for the document to XHTML.
    • Ensures the page is recognized as a valid XHTML document by browsers and tools.

  • xmlns:h="jakarta.faces.html"

    • Defines a namespace with the prefix h for Jakarta Faces HTML tags.
    • Allows you to use JSF tags like <h:form>, <h:inputText>, <h:commandButton>, etc.
    • These tags are part of JSF’s component library and are rendered into standard HTML at runtime.

Why Use Namespaces?

Namespaces prevent name conflicts between standard HTML elements and JSF (or other libraries) tags.
For example: <h:form> (JSF) vs. <form> (plain HTML)

Now your ready to add PrimeFaces, but before we do, let's first discuss a special periodic behavior of NetBeans.

What is "unpacking index for central repository"?

When NetBeans downloads and indexes the Maven Central Repository, it’s essentially building a local catalog of what’s available in that massive online library of Java artifacts. Maven Central contains hundreds of thousands of libraries, plugins, and frameworks—each with its own versions and metadata. Instead of querying the internet every time you type a dependency or browse for artifacts, NetBeans maintains a local index that lets it autocomplete group IDs, artifact IDs, and versions almost instantly.

The process works like this: NetBeans fetches a compressed index file that lists all available artifacts and their coordinates (like groupId:artifactId:version). It then unpacks and parses that data so the IDE can recognize which dependencies exist, show their descriptions, and even offer direct links to documentation or source JARs. In short, this periodic indexing keeps NetBeans in sync with the ever-growing Maven ecosystem—so when you add a dependency to your pom.xml, the IDE knows exactly where to find it and how to help you manage it efficiently.

Why is this important?

Well simply put, this is necessary in order to use the NetBeans GUI to search for dependencies like PrimeFaces and add them automatically. Alternatively, dependencies that exist in the Maven Central Repository can also be added manually by entering their artifact information in the pom.xml.

Example:

<!-- added inside the <dependencies> tags in pom.xml -->
<dependency>
    <groupId>org.primefaces</groupId>
    <artifactId>primefaces</artifactId>
    <version>15.0.0</version>
    <classifier>jakarta</classifier>
</dependency>

And there you have it...

Your project is now ready for Beans, POJOs, JSF pages, and etc. In fact, at this point, you can clean, build, and deploy the project, but of course, it won't do anything.

Speaking of which, before we go, let's talking about a couple of other simple commands that you may ordinarily allow NetBeans to do for you.

As we discussed earlier, Maven is a build tool with a long list of commands that automate the build process for your project. In NetBeans, you can use some of these commands by simply pressing the ▶️ button in the Window, but you may be interested to know that NetBeans is merely running those commands for you which can always be run manually in terminal.

Example:

mvn clean package

This will build a WAR file that can be deployed to your GlassFish server.

Speaking of your GlassFish server, it can be run manually as well. To do this, you would need to locate the asadmin executable. For example, I installed GlassFish using the Homebrew package manager, so I can run my GlassFish server like so...

/opt/homebrew/Cellar/glassfish/7.0.24/libexec/bin/asadmin start-domain --verbose domain1

Then deploying an application is as simple as locating the WAR file and executing the deploy command. For example, if I navigate in terminal to the root folder of a project named Demo I can use...
/opt/homebrew/Cellar/glassfish/7.0.24/libexec/bin/asadmin deploy target/Demo-1.0-SNAPSHOT.war

Now I can visit my web application at http://localhost:8080/Demo-1.0-SNAPSHOT (or whatever the URL is in the glassfish-web.xml file)

Conclusion:

If you've made it this far, hopefully you now have a much better understanding of what NetBeans is doing when you create the basic Maven Web Application. Other IDEs will not use the same archetypes or create all of the same files for you, so this information could be very useful in creating a familiar project in a foreign IDE. It's also quite common to provided instructions for entering command line commands for all sorts of projects, so it is a very good idea to become accustomed to doing so.
